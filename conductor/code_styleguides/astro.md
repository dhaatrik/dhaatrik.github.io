# Astro Component Style Guide

This document outlines key rules, patterns, and best practices for developing Astro components and page layouts in this project.

## 1. File Structure and Naming

- **Component File Naming:** Use `UpperCamelCase` for all component and layout files (e.g., `Header.astro`, `BlogPost.astro`, `TechMarquee.astro`). Page files in `src/pages` should follow standard routing names (e.g., `index.astro`, `about.astro`).
- **File Encoding:** UTF-8.
- **Indentation:** Indent by 4 spaces (or match Prettier config).

## 2. Frontmatter Script (`---` Block)

- **Imports:** Group and organize imports logically:
    1. External/framework imports (e.g., `import { getCollection } from 'astro:content';`).
    2. Astro layouts and components (e.g., `import Header from './Header.astro';`).
    3. Data files, constants, and utilities (e.g., `import { SITE_TITLE } from '../consts';`).
- **TypeScript:** Enforce strict type definitions for component properties by defining a `Props` interface:
    ```astro
    ---
    interface Props {
        title: string;
        description?: string;
    }
    const { title, description } = Astro.props;
    ---
    ```
- **Data Querying:** Use Astro Content Collections APIs (`getCollection`, `getEntry`) rather than loading raw markdown files directly.

## 3. Template HTML Semantics

- **Valid and Semantic HTML:** Follow standard HTML rules (use `<nav>` for navigation, `<footer>` for footer, `<main>` for main content).
- **No Self-Closing Tags for Non-Void Elements:** Ensure custom tags and Astro components always have explicit closing tags (e.g., `<Header />` or `<Header></Header>`).

## 4. Scoped vs. Global Styles

- **Scoped Styles:** Prefer writing CSS rules inside standard `<style>` blocks. Astro automatically scopes these to the component.
- **Global Styles (`is:global`):** Use `<style is:global>` only when styling elements that are generated by markdown rendering, external packages (like KaTeX), or when styles must cascade down to unmanaged children.
- **Tailwind Integration:** Utilize Tailwind CSS classes for structure, spacing, layout, and colors. Avoid using ad-hoc class styling rules; design layout classes consistently.

## 5. Client-Side Scripting (`<script>`)

- **Astro Transition Lifecycle:** Since the site utilizes client-side navigation (View Transitions), register interactive client-side logic on page-load event hooks:
    ```javascript
    <script>
        document.addEventListener('astro:page-load', () => {
            // Initialize proximity trackers, sliders, or canvas animations here.
        });
    </script>
    ```
- **Memory Leaks and Event Cleanup:** Always hook into `astro:before-preparation` to clean up event listeners, timers, intervals, and observers. This prevents duplication and memory leaks when navigating backwards and forwards (bfcache):
    ```javascript
    document.addEventListener(
        'astro:before-preparation',
        () => {
            // Clear timers and remove window/document-level event listeners
            clearInterval(myInterval);
            window.removeEventListener('mousemove', mouseTracker);
        },
        { once: true }
    );
    ```
- **Declarative Web Primitives:** Prefer browser-native primitives (like the native HTML Popover API, CSS Anchor Positioning, Dialog, and CSS Subgrid) for interactive overlays rather than importing third-party JavaScript libraries.

## 6. Accessiblity (a11y)

- **Interactive Targets:** Ensure buttons, links, and popovers have unique, descriptive labels (e.g., `aria-label`, `aria-labelledby`, `aria-describedby`).
- **Focus Management:** Style focus indicators clearly on all interactive elements via `:focus-visible` or Tailwind focus utilities.
